<decisions_made>

- **ntfy.sh over OSC escape sequences**: OSC 777/9 written to /dev/pts/* doesn't work
  in VS Code Remote SSH. Hook stdout goes to Claude Code protocol, not the terminal.
- **`code -r .` over vscode:// URL for window focus**: URL scheme is unreliable with
  multiple remote windows. `code` CLI uses VSCODE_IPC_HOOK_CLI socket for exact targeting.
- **workspace_root in config over cwd detection**: Claude Code cwd drifts, extensionHost
  cwd drifts, VS Code workspace root is client-side. Config is the only reliable source.
- **Pure JS over shell script**: Eliminates curl dependency, bash quoting issues, and
  makes the hook a single portable file.
- **JSON config over shell-sourceable**: Easier to parse in Node.js, supports nesting.
- **Notification title from workspace_root, not cwd**: Prevents drift-induced mismatches
  between notification title and actual window.
</decisions_made>

<context>
The project grew organically from debugging a notification hook in the home directory session.
The prototype works but has the brittleness of a first pass. The core architecture is sound
(ntfy + code CLI focus), but the edge cases around VS Code window targeting, session type
detection, and cwd drift need systematic treatment.

The owner (Logan) uses VS Code Remote SSH from a MacBook (apollo) to a Linux dev server
(dionysus) via Tailscale. He runs tmux inside VS Code's integrated terminal. He also accesses
from iPhone (orpheus) via SSH. The notification system needs to work across all these access
patterns.

Key environment vars available in hooks:
- VSCODE_IPC_HOOK_CLI: /run/user/1000/vscode-ipc-<uuid>.sock (identifies VS Code window)
- TERM_PROGRAM: "tmux" (when in tmux)
- TMUX: set when in tmux session
- SSH_CONNECTION: set when connected via SSH
- CLAUDECODE: "1" (set by Claude Code)
</context>
